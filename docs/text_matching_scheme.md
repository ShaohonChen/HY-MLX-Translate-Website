# 翻译文本段落与句子级高亮匹配方案

## 功能目标

在翻译页面中，用户在「输入文本」或「翻译结果」中选择一段文字时，页面自动在另一侧高亮对应的段落或句子，帮助用户理解原文与译文之间的对应关系。

本功能只实现段落级和句子级对应，不实现词级、短语级或语义相似度匹配。当前翻译模型的输出基本保持原句对应关系，因此采用顺序匹配可以在实现复杂度较低的情况下满足主要使用场景。

## 设计思想

### 结构优先

原文和译文都先被切分为段落，再在段落内切分为句子。匹配时不尝试理解语义，而是利用文本的自然阅读顺序建立对应关系：

- 原文第 `i` 段对应译文第 `i` 段
- 同一段内，原文第 `j` 句对应译文第 `j` 句
- 用户选中多句时，对应另一侧的同一段连续句子区间

### 保守高亮

高亮的目标是辅助阅读，而不是给出绝对精确的语义对齐。只要匹配关系不稳定，就不显示高亮，避免误导用户。

以下情况应当不高亮：

- 另一侧文本为空
- 找不到对应段落
- 找不到对应句子
- 翻译结果仍停留在一个无法判定句子边界的半句中

### 面向流式翻译

翻译结果是流式生成的，因此匹配逻辑必须接受「译文只生成了一部分」的状态。正在生成时，只使用当前已经生成出来的译文内容建立索引，不提前假设未来内容。

当译文只生成到某个句子的一半时，这个半句可以显示在页面中，但不参与句子级匹配。只有句子结束标点出现后，才把它视为一个完整句子。这样可以避免用户选中原文第 `n` 句时，把一个尚未完整生成的译文片段误认为对应句。

## 页面实现方式

当前页面两侧内容使用 `textarea` 展示。原生 `textarea` 不能在内部插入局部高亮标签，因此需要为每个文本框增加一个高亮镜像层。

建议结构如下：

```html
<div class="text-match-box">
  <div id="sourceHighlight" class="highlight-layer"></div>
  <textarea id="sourceText"></textarea>
</div>

<div class="text-match-box">
  <div id="resultHighlight" class="highlight-layer"></div>
  <textarea id="resultText" readonly></textarea>
</div>
```

实现要点：

- `textarea` 继续负责输入、选择、滚动和复制
- `highlight-layer` 只负责显示高亮背景
- 两层使用相同的字体、字号、行高、内边距和换行规则
- `textarea` 背景设为透明或半透明，让下方高亮层可见
- 两侧滚动时同步对应高亮层的 `scrollTop`
- 翻译结果应使用 `readonly`，不要使用 `disabled`，否则用户不方便选择结果文本

## 文本切分规则

### 段落切分

段落以连续换行作为边界：

- 一个或多个换行符可视为段落分隔符
- 空段落不参与匹配
- 每个段落记录其在完整文本中的 `start` 和 `end` 偏移量

段落匹配采用顺序一对一规则。原文和译文段落数不一致时，只匹配两侧都存在的段落。

### 句子切分

句子切分在段落内部进行，优先使用浏览器原生能力：

```js
new Intl.Segmenter(locale, { granularity: 'sentence' })
```

如果浏览器不支持 `Intl.Segmenter`，则使用标点规则兜底：

- 中文：`。！？`
- 英文：`.?!`
- 日文、韩文及其他语言：兼容上述中英文标点

句子对象需要记录：

- 所属段落索引
- 段内句子索引
- 在完整文本中的 `start` 和 `end`
- 是否为完整句子

正在流式翻译时，译文最后一个没有结束标点的片段标记为不完整句子，不参与句子级匹配。翻译完成后，如果最后一句没有结束标点，可以把它作为完整句子参与匹配，因为此时模型已经停止生成。

## 选区识别规则

用户选择文本时，通过 `selectionStart` 和 `selectionEnd` 获取选区范围。

### 单句选择

只要用户选区与某个句子有重叠，就认为用户选中了这个完整句子，而不是只选中了其中几个字符。

例如用户只选中了句首几个字，或为了方便拖拽没有严格覆盖句尾标点，也应该把整句作为匹配单位。

### 多句选择

如果用户一次选择了 2、3 句话，或者选区跨过多个句子的一部分，则将所有与选区有重叠的句子都纳入匹配范围。

处理规则：

- 找出选区所在段落
- 找出该段落内所有与选区相交的句子
- 将这些句子扩展为完整句子区间
- 在另一侧对应段落中高亮相同句子索引范围

示例：

- 原文选区碰到了第 2、3、4 句
- 则译文高亮第 2、3、4 句
- 即使选区只碰到第 2 句的后半部分和第 4 句的前半部分，也视为完整选中了第 2 到第 4 句

### 跨段落选择

跨段落选择先按段落级处理。只要用户选区触碰了多个段落，就高亮另一侧对应的连续段落范围。

这样可以避免跨段落时再做复杂的句子区间拼接，同时更符合用户大范围选择时的阅读意图。

## 匹配流程

```text
用户在一侧选择文本
        |
        v
读取 selectionStart / selectionEnd
        |
        v
判断选区触碰到的段落范围
        |
        +-- 触碰多个段落 --> 映射为另一侧连续段落高亮
        |
        +-- 只触碰一个段落
                |
                v
        找出该段落内所有与选区相交的句子
                |
                +-- 找不到完整句子 --> 不高亮
                |
                +-- 找到句子区间
                        |
                        v
                映射到另一侧同段落、同句子索引区间
                        |
                        +-- 另一侧句子尚未完整生成 --> 不高亮或只高亮已完整句子
                        |
                        +-- 匹配成功 --> 渲染高亮
```

## 流式生成中的处理策略

流式生成时最容易出现的问题是：用户选择了原文某一句，但译文只生成到对应句子的一半。此时如果强行高亮，用户会误以为半句话就是完整译文。

因此采用以下策略：

1. 翻译过程中持续更新译文索引，但最后一个未结束句子标记为 `isComplete: false`
2. 句子级映射只使用 `isComplete: true` 的译文句子
3. 如果目标句子范围中包含尚未完整生成的句子，则不高亮该未完成句子
4. 如果用户选择的是已经完整生成的前几句，可以正常高亮
5. 翻译完成后，重新构建索引，并允许最后一个无结束标点的句子参与匹配

这样可以兼顾两个目标：

- 翻译过程中，已经稳定生成的句子可以立即高亮
- 生成到一半的句子不会被错误标记为完整对应关系

如果用户点击「取消翻译」，当前结果应视为一次已结束但可能不完整的输出。取消后可以重建索引，并将最后一个无结束标点的句子标记为完整。原因是此时不会再继续生成，用户看到的就是最终保留下来的译文。

## 高亮渲染规则

高亮层根据另一侧文本和待高亮范围重新渲染 HTML：

- 普通文本必须进行 HTML 转义
- 高亮范围用 `<mark>` 包裹
- 多个范围需要先排序并合并重叠区间
- 没有有效范围时清空高亮层

高亮样式建议：

```css
.highlight-layer mark {
  background: rgba(255, 230, 120, 0.75);
  border-radius: 2px;
}
```

交互规则：

- 用户在输入文本中选择时，高亮翻译结果
- 用户在翻译结果中选择时，高亮输入文本
- 取消选择、点击空白区域、重新翻译、清空文本时移除高亮
- 翻译流式追加内容时，如果当前选区仍然有效，可以重新计算并刷新高亮

## 缓存与性能

文本索引只需要在文本内容变化时重建：

- 输入文本变化时，重建输入文本索引
- 译文流式追加时，重建译文索引
- 翻译完成或取消后，使用完成态规则重建译文索引

选择事件需要做轻量防抖：

- `selectionchange`
- `mouseup`
- `keyup`
- `touchend`

建议延迟 80-120ms 后再执行匹配，避免拖拽选择过程中频繁计算。

对于当前项目的文本规模，段落和句子切分的计算量很小。即使每次流式追加后重建译文索引，也不会成为主要性能瓶颈。

## 边界情况

| 场景 | 处理方式 |
| --- | --- |
| 另一侧文本为空 | 不高亮 |
| 翻译正在生成 | 只匹配已完整生成的句子 |
| 翻译生成到半句 | 半句不参与句子级匹配 |
| 翻译被用户取消 | 将当前结果视为结束态，允许最后半句参与匹配 |
| 用户选中一句的一部分 | 扩展为完整句子 |
| 用户选中多句的一部分 | 扩展为连续完整句子区间 |
| 用户跨段落选择 | 按连续段落范围高亮 |
| 段落数不一致 | 只匹配两侧都存在的段落 |
| 句子数不一致 | 只匹配两侧都存在的句子 |
| 匹配范围不存在 | 不高亮 |

## 后续可扩展方向

当前方案不引入模型侧改动，也不修改翻译 prompt。后续如果需要更精确的对齐，可以考虑：

- 让模型输出隐藏句子编号或锚点，再在前端隐藏这些锚点
- 在翻译完成后做一次句向量相似度匹配
- 为长文档增加段落折叠和双栏同步滚动

这些扩展会增加实现复杂度，也可能影响翻译输出格式。当前版本优先保持简单、稳定、可解释。

## 总结

本方案采用「段落顺序匹配 + 句子顺序匹配 + 保守高亮」的实现路线。它不追求复杂的语义对齐，而是利用当前翻译模型稳定的一句对一句输出特点，为用户提供足够可靠的阅读辅助。

核心实现重点是：

- 使用镜像高亮层解决 `textarea` 无法局部高亮的问题
- 选择到句子的一部分时，自动扩展为完整句子
- 选择到多句时，高亮连续句子区间
- 流式生成中只匹配已经完整生成的句子
- 翻译完成或取消后，按最终文本重新建立匹配索引
